# Lawang (BPS Semarang) — Architecture & Code-Quality Audit

**Date:** 2026-06-13 · **Scope:** Architecture + maintainability, code-quality bugs (per request).
**App:** Flutter, ~30k lines Dart, 25 screens, data fetched from GitHub raw JSON + 116 source `.xlsx` files. **Zero tests.**

---

## Summary

| Metric | Before | After fixes |
|---|---|---|
| Analyzer **errors** | 80 | 49 (all in dead code) |
| Live-code errors | 31 | **0** |
| Analyzer **info** lints | 977 | ~975 |

The 80 `error`-level findings come from `strict-casts: true` in `analysis_options.yaml`. They compile and usually run (the Dart front-end inserts runtime checks), but **each is a latent `TypeError` crash** if the remote GitHub JSON shape drifts (e.g. a number arrives as a string, or a field is null). Since the data is fetched from an external repo that can change independently of the app, this is the top risk class.

**Fixed this pass (31 live-code crash risks):** hardened every JSON read in `tenaga_kerja_screen`, `penduduk_screen`, `ipm_screen`, `sdgs_data_service` to use `(x as num?)?.toDouble()/.toInt() ?? default` and `?.toString()` at the parse boundary — mirrors the already-correct pattern in `ekonomi_data.dart`.

## Findings (severity ordered)

### 🔴 C1 — Dead code holds 49 of 49 remaining errors *(recommend delete)*
Four files are imported/referenced by **nothing** (verified by reachability scan from `main.dart`):
- `lib/statistics_data.dart` (16 err) — `StatisticsData`/`DataPoint`/`Province`, zero refs.
- `lib/map_screen.dart` (16 err) — whole "peta" feature, `MapScreen` never instantiated.
- `lib/data_list_screen.dart` (7 err) — `DataListScreen` never instantiated.
- `lib/data_detail_screen.dart` (10 err) — only imported by `data_list_screen` (dead chain).

Deleting these removes 100% of remaining analyzer errors and ~2,600 lines of unmaintained code. **Held back pending your confirmation** — `map_screen` is a sizeable feature that may be parked intentionally.

### 🔴 C2 — No model/parse layer → dynamic-map access everywhere *(root cause)*
Most screens read `data['field']` (type `dynamic`) inline in `build()` and pass it straight to typed widgets (`fl_chart` wants `double`). This is the source of all 80 errors. `ekonomi_data.dart` is the only file doing it right (`(json['year'] as num).toInt()`). **Fix direction:** introduce typed model classes with `fromJson` per category (like `EkonomiData`), parse once, pass typed objects to widgets.

### 🟠 H1 — Duplicate, redundant data-fetch stacks
Two parallel GitHub layers hit the **same** base URL and `version.txt`:
- `services/github_data_service.dart` + `github_data_repository.dart` (10 categories)
- `sdgs_github_repository.dart` + `sdgs_data_service.dart` (SDGs only)

Result: SDGs version/data fetched twice, two separate caches, two cache-key schemes. Consolidate SDGs into `GitHubDataService` (it already lists `'sdgs'` as a category).

### 🟠 H2 — All-static singletons → untestable, zero tests
`GitHubDataService`, `EkonomiDataManager`, `SDGsDataService` are static/global. No dependency injection, so screens can't be unit-tested with fake data. Combined with **no test directory contents**, every regression is caught only by manual run. Recommend extracting an injectable repository interface and adding widget/parse tests at the model boundary.

### 🟡 M1 — `AppInitializer` success path never navigates (`main.dart:73-84`)
`_initializeApp()` navigates to `/home` **only in the `catch` block**. On success it relies entirely on `SplashScreen`'s internal 8s timer to advance. Works today but is fragile coupling; the error path can also double-`pushReplacementNamed` against the splash's own navigation. Make navigation explicit and single-sourced.

### 🟡 M2 — Monolithic "God" screens
`inflasi_screen` 2673, `pendidikan_screen` 2075, `penduduk_screen` 2020, `ipg_screen` 1898 lines. Each mixes parsing + business logic + chart config + layout in one `State`. Extract widgets and a per-category controller/model.

### 🟢 L1 — 977 info lints
`deprecated_member_use` 320 (mostly `Color.withOpacity` → `.withValues`), `avoid_redundant_argument_values` 185, `prefer_int_literals` 152, `prefer_const_constructors` 122. None are bugs; `withOpacity` will break on a future Flutter major. Bulk-fixable with `dart fix --apply`.

### 🟢 L2 — 26 raw `print()` in production paths
`main.dart`, `splash_screen.dart` log via `print` (unguarded). `avoid_print` is disabled in config. Switch to `debugPrint` / a logger guarded by `kDebugMode`.

## Recommended next steps
1. **Confirm dead-code deletion** (C1) → clears all errors.
2. Build typed models per category (C2) → permanently prevents the dynamic-cast bug class.
3. Merge the two GitHub stacks (H1).
4. `dart fix --apply` for the info lints (L1/L2), then re-enable stricter CI gating.
5. Add parse-boundary + widget tests once models exist (H2).
